# 移除 LLM 索引分析入口和占位能力

Status: ready-for-agent

## Problem Statement

项目未来不考虑接入 LLM，但当前两个索引分析 MCP 工具仍把 `llm` 宣传为可选分析方法。PostgreSQL、MySQL、Doris 和达梦方言也保留了识别该方法并返回“暂未接入”说明的占位分支，对应测试和兼容性文档继续暗示未来可能提供 LLM 能力。

这些入口会让 MCP 客户端误以为项目具备或计划具备 LLM 索引分析能力，增加无效的 API 表面积、条件分支、测试和维护成本。同时，项目仍然需要 Spring AI 提供的 MCP Server 基础设施，不能因为依赖或包名中包含 `spring-ai` 就误删 MCP 服务能力。

## Solution

保留 MCP Server、现有 MCP 工具、数据库诊断能力和基于 `dta`/规则引擎的索引分析，只移除所有对 `method='llm'` 的宣传、识别分支、占位响应、专用测试和面向用户的占位文档。

两个索引分析工具继续保留 `method` 参数以兼容现有 `dta` 调用。空值或空白值继续按默认的 `dta` 方法处理；除 `dta` 外的任何方法都在统一的 MCP 工具边界被拒绝，不进入具体数据库方言，也不触发 SQL 查询。错误信息应描述“仅支持 dta”，不承诺未来增加其他分析方法。

## User Stories

1. 作为 MCP 客户端使用者，我希望工具说明只列出实际可用的分析方法，以便正确构造调用参数。
2. 作为 MCP 客户端使用者，我希望 `dta` 索引分析保持可用，以便此次清理不会破坏现有工作流。
3. 作为 MCP 客户端使用者，我希望省略 `method` 时仍使用默认分析方法，以便已有调用继续兼容。
4. 作为 MCP 客户端使用者，我希望传入空白 `method` 时得到与默认方法一致的行为，以便不同客户端的可选参数序列化方式不会造成回归。
5. 作为 MCP 客户端使用者，我希望传入不支持的方法时收到明确错误，以便快速修正调用而不是得到误导性的占位结果。
6. 作为 MCP 客户端使用者，我希望两个索引分析工具继续存在，以便工具发现结果和既有集成保持稳定。
7. 作为 MCP 客户端使用者，我希望其他 MCP 工具保持不变，以便此次变更只影响被废弃的分析方法入口。
8. 作为数据库管理员，我希望不支持的方法在任何数据库查询前被拒绝，以便无效请求不产生数据库负载。
9. 作为数据库管理员，我希望 PostgreSQL 的规则索引建议保持原有扩展检查和分析流程，以便现有诊断能力不受影响。
10. 作为数据库管理员，我希望 MySQL 的规则索引建议保持原有 Performance Schema 检查和评分流程，以便现有诊断能力不受影响。
11. 作为数据库管理员，我希望 Doris 的只读索引建议保持原有行为，以便现有诊断能力不受影响。
12. 作为数据库管理员，我希望达梦的只读索引建议和 SQL 安全校验保持原有行为，以便现有诊断能力不受影响。
13. 作为项目维护者，我希望所有数据库方言遵循同一个方法校验规则，以便不再维护重复的 LLM 判断分支。
14. 作为项目维护者，我希望公共诊断辅助逻辑中不再存在 LLM 专用判断，以便代码表达项目真实能力。
15. 作为项目维护者，我希望测试不再断言占位响应，以便测试套件只覆盖受支持的产品行为。
16. 作为项目维护者，我希望 README 只描述当前支持的能力，以便新用户不会把占位入口当成路线图承诺。
17. 作为项目维护者，我希望兼容性文档不再宣称保留 LLM 入口，以便各数据库能力矩阵与实际实现一致。
18. 作为项目维护者，我希望 MCP Server 仍能正常启动并发现原有工具，以便清理不会误删基础设施。
19. 作为项目维护者，我希望继续保留 MCP Server 所需的 Spring AI 依赖，以便依赖名称不会导致错误的删除范围。
20. 作为后续实现代理，我希望规格明确列出删除范围和保留范围，以便不会扩张为 MCP 架构重写。
21. 作为代码审查者，我希望变更后生产代码、测试和用户文档不再包含 LLM 占位实现或未来接入承诺，以便确认清理完整。
22. 作为代码审查者，我希望权限校验仍在查询索引分析之前执行，以便本次 API 清理不会削弱已有安全边界。

## Implementation Decisions

- 保留 Spring Boot 应用、Spring AI MCP Server starter、MCP 注解和 MCP 工具注册；这些属于 MCP 基础设施，不属于待删除的 LLM 能力。
- 保留 `analyze_workload_indexes` 和 `analyze_query_indexes` 两个 MCP 工具的名称、用途和现有非 LLM 参数。
- 保留两个工具的 `method` 参数，唯一受支持的显式值为 `dta`；空值和空白值规范化为 `dta`，以保持当前默认行为。
- 在 MCP 工具门面统一完成分析方法规范化和校验。任何非 `dta` 值都返回明确的参数错误，并且不得调用数据库方言或执行 SQL。
- MCP 参数说明只描述 `dta` 和默认行为，不再列出 LLM、占位能力或未来接入计划。
- 数据库诊断接口继续接收规范化后的分析方法，避免扩大公共接口变更；具体方言只负责已支持的规则分析流程。
- PostgreSQL、MySQL、Doris 和达梦方言删除 LLM 分支及占位消息生成逻辑，不新增替代占位层。
- 公共诊断辅助模块删除只服务于 LLM 判断的辅助函数和相关导入。
- `analyze_query_indexes` 现有权限校验顺序保持不变：每条查询先完成授权，再进入索引分析。方法校验应位于统一工具边界，并保证无效方法不会进入数据库方言。
- `analyze_workload_indexes` 不新增用户身份或权限参数；此次变更不调整其既有权限模型。
- 所有数据库原有的查询数量限制、只读 SQL 校验、扩展或功能可用性检查、大小预算、执行计划分析和降级信息保持不变。
- README 和数据库兼容性说明删除 LLM 入口、占位返回以及未来接入表述；支持能力统一描述为 `dta` 或当前数据库方言实际使用的规则引擎。
- 不因依赖名称中出现 `spring-ai` 而删除依赖管理或 MCP Server starter。
- 完成后，除本规格作为历史需求说明外，生产代码、测试和用户文档不应再包含 LLM 专用入口、占位响应或未来接入承诺。

## Testing Decisions

- 主要测试接缝是 MCP 工具门面，因为所有索引分析 MCP 调用都经过这一边界。测试应验证默认方法规范化、`dta` 转发和未知方法拒绝等外部可观察行为，不直接断言私有辅助函数或分支结构。
- 在工具门面测试中验证：省略或传入空白 `method` 时向当前数据库方言传递 `dta`；显式传入 `dta` 时保持正常转发。
- 在工具门面测试中使用一个中性的未知方法值验证统一拒绝行为，并确认数据库方言和 SQL 客户端均未被调用。该测试覆盖包括旧 `llm` 值在内的所有非 `dta` 输入，无需保留 LLM 专用测试用例。
- 在 MCP 注解和输入 Schema 测试中验证两个工具仍然可发现，工具总数和名称不变，参数说明只宣传 `dta`，权限相关的 `user_id` Schema 属性保持不变。
- 保留各数据库方言现有的 `dta`/规则分析测试，继续覆盖 PostgreSQL、MySQL、Doris 和达梦的受支持路径；删除仅验证 LLM 占位消息的测试。
- 保留达梦只读 SQL 校验测试，但将其与 LLM 方法组合的重复用例移除；SQL 安全行为由受支持方法路径继续证明。
- 使用现有 MCP 注解测试作为工具发现和 Schema 契约的先例，使用现有工具门面权限测试作为跨模块调用顺序的先例，使用各方言诊断测试作为数据库特定行为的先例。
- 运行完整测试套件，确认 MCP 应用上下文仍可启动，九个现有 MCP 工具仍被注册，权限校验和数据库诊断回归测试全部通过。
- 将内容扫描作为完成检查而非行为单元测试：生产代码、测试、README 和兼容性文档中不应残留 LLM 专用标记；本地规格目录可作为需求历史保留相关文字。

## Out of Scope

- 删除或替换 MCP Server 基础设施。
- 删除 Spring AI BOM 或 MCP Server starter。
- 修改 MCP 传输协议、服务端口、工具注册机制或 Spring Boot 启动方式。
- 删除两个索引分析 MCP 工具或移除其 `method` 参数。
- 重新设计 `dta`、规则引擎、候选索引生成、评分或执行计划分析算法。
- 接入任何 LLM、模型供应商、提示词框架、向量数据库或 Agent 推理能力。
- 修改权限校验模块、指标权限规则、Redis 缓存或 `user_id` 契约。
- 修改支持的数据库类型或增加新的数据库方言。
- 改变 SQL 只读限制、查询数量上限、索引大小预算或现有扩展依赖。
- 为旧 `method='llm'` 调用提供兼容占位响应、迁移适配器或弃用期。

- 已确认最高层测试接缝为 MCP 工具门面，并辅以现有的 MCP Schema 测试和数据库方言回归测试。
