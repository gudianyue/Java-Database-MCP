# 指标权限控制改造计划

## 背景

当前 ask-data 场景中的指标权限主要依赖 Agent 在调用 MCP 前自行遵守规则。这个边界不够可靠：Agent 可能遗漏权限字段、误写 SQL、幻觉授权范围，或只做了预检但实际执行了不同 SQL。

本次改造目标是把指标权限控制下沉到 Java Database MCP 服务端，让 MCP 成为强制权限边界。Agent 传入的结构化字段只作为声明，实际 SQL 必须作为证据接受服务端校验；声明与 SQL 不一致、权限 provider 不可用、SQL 无法可靠判断时，都必须 fail-closed 拒绝。

## 改造目标

- 在 MCP 服务端建立指标领域的强制权限边界，不依赖 Agent 自觉。
- 覆盖所有用户可传 SQL 的 MCP 工具：`execute_sql`、`explain_query`、`analyze_query_indexes`。
- 要求指标查询调用显式传入 `permission_domain=metric`、`user_id` 和指标授权范围声明。
- 服务端同时校验结构化声明和 SQL 证据，二者必须一致且在用户授权范围内。
- 指标授权范围使用 `(quota_id, quota_scene)` 元组或等价 map 表达，避免两个独立集合产生交叉误放。
- 通过插件化 provider 支持 Apboa 默认实现，同时保留通用 MCP 的可移植性。
- 在不影响非指标 SQL 的前提下，保护配置中声明的指标表、指标列和场景列。

## 非目标

- 本次不做 MCP 登录鉴权。
- 本次不验证 `user_id` 的可信来源，默认由调用方、网关或上游系统保证。
- 本次不把指标权限强加到所有数据库查询，只保护配置命中的指标领域 SQL。
- 本次不替换现有只读 SQL 安全控制，例如写操作拦截、多语句限制等。
- 本次不把可选预检或调试工具作为安全边界。
- 本次不做高风险 SQL 重写或自动注入行级过滤条件。

## 核心原则

- MCP 是强制权限边界，不能依赖 Agent 提示词、自觉或调用习惯。
- 结构化字段是声明，SQL 是证据，两者必须一致。
- 权限判断必须发生在 SQL 进入执行、解释或分析服务之前。
- 指标 SQL 无法可靠判断时 fail-closed 拒绝。
- provider 异常、超时、缺失或返回异常数据时 fail-closed 拒绝。
- 非指标 SQL 尽量保持原有行为，只继续受现有 SQL 安全控制约束。
- 第一版优先保守、可审查、可测试；宁可误拒复杂 SQL，也不能误放越权 SQL。

## 需要保护的工具范围

必须覆盖所有用户可传 SQL 的 MCP 工具：

- `execute_sql`：执行前必须完成指标权限校验。
- `explain_query`：解释计划前必须完成同样的指标权限校验；`analyze=true` 不能绕过权限控制。
- `analyze_query_indexes`：列表中的每一条用户 SQL 都必须单独校验；任一受保护 SQL 被拒绝或无法判断时，应让工具调用失败并返回脱敏错误。

暂不纳入本轮强制边界的工具：

- 不接收用户 SQL 的健康检查、负载、元数据类诊断工具。
- 未来新增任何接收用户 SQL 的工具时，必须接入同一个权限校验入口。

## 工具参数契约

所有可传 SQL 的工具需要暴露概念等价的权限参数：

- `sql` 或查询列表：用户要执行、解释或分析的 SQL。
- `permission_domain`：指标领域传 `metric`，非指标 SQL 可传 `none` 或使用等价默认值。
- `user_id`：`permission_domain=metric` 时必填。
- `metric_scopes`：推荐结构化列表，每项为 `{ quota_id, quota_scene }`。

如果 MCP 注解或 schema 暂时不方便表达嵌套对象，可使用严格一一配对的并行列表：

- `quota_ids[i]` 与 `quota_scenes[i]` 组成一个授权元组。
- 两个列表长度不一致必须拒绝。
- 禁止把 `quota_ids` 和 `quota_scenes` 解释为两个独立集合并计算笛卡尔积。

工具描述需要明确告诉 Agent：指标 SQL 必须传 `permission_domain=metric`、`user_id`、`metric_scopes`；MCP 会重复校验 SQL 与声明是否一致。

## 权限模型

建议新增通用权限模型包，核心对象包括：

- `PermissionDomain`：例如 `NONE`、`METRIC`。
- `PermissionContext`：包含权限领域、`user_id`、声明的指标范围、工具用途。
- `MetricScope`：一个明确的 `(quota_id, quota_scene)` 元组。
- `PermissionScope`：用户已授权范围，使用元组集合或 `Map<quota_id, Set<quota_scene>>` 表达。
- `PermissionProvider`：按领域解析授权范围的插件接口。
- `ProtectedResourceSpec`：受保护表、指标列、场景列配置。
- `DomainSqlInspector`：从 SQL 中保守识别受保护资源和指标谓词。
- `PermissionDecision`：允许或拒绝，并携带脱敏 reason category。
- `PermissionEnforcer`：统一编排 provider、声明校验、SQL 证据校验和失败处理。

关键约束：授权范围必须保持元组语义。比如用户只拥有 `(A, default)` 和 `(B, custom)`，绝不能因为两个独立集合 `{A, B}` 与 `{default, custom}` 而误放 `(A, custom)`。

## 服务端校验流程

建议统一入口为 `PermissionEnforcer.authorize(sql, context, usageMode)`：

1. 工具层接收 SQL 和权限参数。
2. 构造 `PermissionContext`。
3. SQL inspector 判断是否访问配置中的受保护指标表。
4. 未访问受保护指标表时，按原有非指标流程继续。
5. 访问受保护指标表时，要求 `permission_domain=metric`。
6. `permission_domain=metric` 时要求 `user_id` 和声明的指标元组。
7. provider 根据 `user_id` 解析服务端授权元组。
8. 校验声明的指标元组全部在授权范围内。
9. SQL inspector 提取 SQL 中实际指标和场景谓词。
10. 校验 SQL 证据与结构化声明一致。
11. 任一步骤缺失、异常、不一致或无法可靠判断时拒绝。
12. 只有允许结果才能进入 `SqlClient`、解释计划服务或索引分析服务。

现有 `RestrictedSqlGuard` 继续保留在 SQL 执行链路中，负责通用读写安全；指标权限是更靠前的业务授权层。

## SQL 检查范围

第一版 inspector 只接受可证明安全的 SQL 子集：

- 单查询块 `SELECT` 或 `EXPLAIN SELECT`，允许一个结尾分号。
- 配置中的受保护表，支持 schema-qualified 表名，大小写不敏感。
- 简单 `FROM` / `JOIN` 别名，谓词可引用别名或表名。
- 指标列和场景列的字面量等值谓词。
- 指标列与场景列严格配对的字面量 tuple-IN 谓词，例如 `(quota_id, quota_scene) IN (...)`。
- 使用 `AND` 组合的简单谓词。

第一版应拒绝或标记为不可判断：

- 访问受保护指标表但缺少可识别的指标和场景过滤。
- `WITH`、`UNION`、`INTERSECT`、`EXCEPT`、`MINUS`、`OR` 或 `NOT`。
- 独立的指标 `IN (...)` 与场景 `IN (...)`，以及任何非等值 scope 运算符。
- 子查询、函数、类型转换、字符串拼接、参数占位、计算表达式参与指标谓词。
- 未配置的视图、未限定表名或无法解析的表来源。
- tokenizer 无法可靠区分的注释、字符串字面量、引用标识符边界情况。
- 任何无法证明受保护表访问范围和指标谓词一致的 SQL。

实现上应至少有轻量 tokenizer，先分离注释、字符串字面量、标识符、别名和谓词，再做判断。不能用脆弱的裸正则直接充当 SQL 解析器。

## IN 条件元组语义

`IN` 条件必须保持指标与场景的元组语义。

允许的判断方式：

- SQL 直接表达了可枚举的明确元组。
- inspector 能证明 `quota_id` 与 `quota_scene` 的组合只落在声明和授权的元组集合内。
- 使用等价 map 表达时，按 `quota_id -> allowed quota_scene set` 比较。

禁止的判断方式：

- 从 `quota_id IN ('A', 'B')` 和 `quota_scene IN ('default', 'custom')` 推导出四个组合后直接放行。
- 把授权的指标 ID 集合和场景集合拆开独立校验。
- 在无法证明 SQL 组合关系时猜测 Agent 意图。

如果 SQL 的 `IN` 组合无法可靠证明只包含授权元组，应 fail-closed 拒绝，并返回类似 `permission_sql_uninspectable` 或 `permission_sql_mismatch` 的脱敏原因。

## 插件化设计

通用 MCP 核心只定义权限扩展点，不硬编码 Apboa 业务概念：

- provider 接口按 `PermissionDomain` 注册。
- 受保护资源、指标列、场景列来自配置。
- provider 负责把项目授权数据归一化为通用 `MetricScope`。
- enforcer 只依赖通用接口和配置，不依赖 Apboa 表结构。
- 插件关闭时，非指标 SQL 保持现有行为；指标插件启用后，命中受保护资源必须执行强制校验。

可预留用法模式 `usageMode`：

- `EXECUTE`：对应 `execute_sql`。
- `EXPLAIN`：对应 `explain_query`。
- `ANALYZE_INDEX`：对应 `analyze_query_indexes`。

不同模式可以共享授权逻辑，只在错误文案和下游入口上区分。

## Apboa 默认插件

Apboa 默认插件建议提供开箱配置和 provider：

- 默认受保护表包括 ask-data 常用指标事实表、计划表、指标基础表和公开指标汇总表。
- 默认指标列可配置为 `quota_id`、`ind_id`、`ind_code` 等等价列。
- 默认场景列可配置为 `quota_scene`、`scene_name` 等等价列。
- 授权来源使用参数化 SQL，通过 `user_id` 查询授权的 `quota_id` 与 `quota_scenes`。
- 如果授权行中的 `quota_scenes` 包含多个场景值，provider 必须按 Apboa 语义拆分并归一化为明确元组集合。

Apboa 默认插件只能作为默认实现，不应污染通用 `DatabaseToolFacade`、`SqlClient` 或核心权限模型。

## 配置项建议

建议在 `database-mcp` 配置根下增加：

```yaml
database-mcp:
  permission:
    enabled: true
    metric:
      enabled: true
      protected-tables:
        - gkschema.gk_qta_data
        - gkschema.gk_qta_data_plan
        - gkschema.gk_qta_info_base
        - twb_data.dm_ind_fi_pub_bil
        - twb_data.dm_ind_fm_pub_bil_sum
      metric-columns:
        - quota_id
        - ind_id
        - ind_code
      scene-columns:
        - quota_scene
        - scene_name
      provider:
        authorization-query: "<parameterized SQL>"
        timeout-seconds: 10
```

实际表名、列名和授权 SQL 以部署项目配置为准。默认值应便于 Apboa 落地，但允许其他项目替换 provider 和资源配置。
启用指标权限时，受保护表、指标列、场景列必须为非空，并且 Spring 上下文中必须恰好存在一个 `MetricPermissionProvider`。授权 SQL 必须包含且只包含一个用户参数：推荐使用 `${user_id}` 或 `${p_user_id}`，也可使用单个 `?`；该参数始终绑定当前 `user_id`。`timeout-seconds` 必须大于 0。

## 可选预检/调试工具

可选增加以下 MCP 工具，帮助 Agent 规划和排错：

- `check_metric_permission`：检查某个 `user_id` 对单个 `(quota_id, quota_scene)` 是否有权限。
- `list_authorized_metrics`：列出某个 `user_id` 的授权指标范围。
- `explain_metric_permission`：开发或调试环境解释拒绝原因。

这些工具只能作为辅助信息，不能作为安全边界。`execute_sql`、`explain_query`、`analyze_query_indexes` 在实际处理 SQL 时必须重复执行完整权限校验。

## 错误处理

所有拒绝应返回脱敏、稳定的错误类别，避免泄露完整授权范围或敏感 SQL 细节：

- `permission_denied`
- `permission_context_missing`
- `permission_sql_mismatch`
- `permission_sql_uninspectable`
- `permission_provider_unavailable`
- `permission_provider_timeout`
- `permission_plugin_disabled_or_missing`

provider 异常、超时、缺失、返回格式异常时全部 fail-closed。日志可以记录 request id、工具名、原因类别和必要的诊断上下文，但不要在工具响应或普通日志中输出用户完整授权元组集合。

## ask-data Skill 同步修改

后续实现时需要同步修改 ask-data skill，使其明确要求：

- 指标领域 MCP 调用必须传 `permission_domain=metric`。
- 必须传 `user_id`。
- 必须传 `metric_scopes`，即明确的 `(quota_id, quota_scene)` 元组列表。
- SQL 中的指标和场景过滤必须与 `metric_scopes` 一致。
- 不得把预检工具结果当作执行安全证明。
- 非指标元数据 SQL 只有在不触碰受保护指标表时，才可使用 `permission_domain=none`。
- 预期 MCP 会拒绝声明与 SQL 不一致、缺少上下文、越权或无法检查的 SQL。

本计划只记录同步要求；具体文件修改在实施阶段执行。

## 实施步骤

1. 梳理当前工具入口，确认 `execute_sql`、`explain_query`、`analyze_query_indexes` 的 SQL 入参和下游调用链。
2. 增加权限配置绑定，支持全局开关、指标插件开关、受保护表、指标列、场景列和 provider 配置。
3. 增加通用权限模型：`PermissionDomain`、`PermissionContext`、`MetricScope`、`PermissionDecision` 等。
4. 增加 `PermissionProvider` 接口和无可用 provider 时的 fail-closed 行为。
5. 实现 Apboa metric provider，把授权 SQL 结果归一化为明确 `(quota_id, quota_scene)` 元组。
6. 实现轻量 SQL tokenizer 和 conservative inspector，支持第一版可证明安全的 SQL 子集。
7. 实现 `PermissionEnforcer`，串联受保护资源检测、上下文校验、provider 授权、声明校验和 SQL 证据校验。
8. 将 enforcer 接入 `execute_sql`，确保授权通过后才进入 `SqlClient`。
9. 将 enforcer 接入 `explain_query`，确保 explain 和 analyze explain 都不能绕过权限。
10. 将 enforcer 接入 `analyze_query_indexes`，逐条校验传入 SQL。
11. 更新 MCP 工具注解、描述和参数 schema，让 Agent 能看到权限字段契约。
12. 如有需要，增加可选预检和调试工具，并在描述中声明它们不是安全边界。
13. 同步更新 ask-data skill 的调用要求。
14. 运行单元测试、集成测试、注解测试和现有 SQL guard 回归测试。

## 测试计划

单元测试：

- 授权元组模型保持 `(quota_id, quota_scene)` 语义。
- `(A, default)` 与 `(B, custom)` 不允许误放 `(A, custom)`。
- 声明缺失 `permission_domain=metric` 时拒绝受保护指标 SQL。
- 缺失 `user_id` 时拒绝。
- 声明范围不在 provider 授权范围内时拒绝。
- SQL 谓词与声明元组不一致时拒绝。
- protected table 出现在注释或字符串字面量中时不产生误判。
- 无法可靠解析的指标 SQL 返回 fail-closed。
- provider 异常、超时、缺失时拒绝。

集成测试：

- `execute_sql` 对授权且声明与 SQL 一致的指标查询放行。
- `execute_sql` 对未授权、缺上下文、声明不一致、SQL 不可检查的指标查询拒绝。
- `explain_query` 对受保护指标 SQL 使用同样权限规则。
- `analyze_query_indexes` 对查询列表逐条校验，任一受保护 SQL 不合规则拒绝。
- 插件关闭或非指标 SQL 场景保持原有通用行为。
- 现有 `RestrictedSqlGuard` 相关测试继续通过。

注解和契约测试：

- MCP 工具 schema 暴露 `permission_domain`、`user_id`、`metric_scopes` 或等价字段。
- 工具描述明确声明指标 SQL 的权限字段要求。
- 可选预检工具描述明确不能绕过执行期校验。

回归验证：

- 运行项目现有测试套件。
- 针对权限模块运行新增测试。
- 人工审查 ask-data skill 文案，确认不再暗示 Agent 预检即可绕过执行期校验。

## 风险与缓解

- SQL 检查误放：采用保守 inspector，只放行可证明安全的 SQL 子集。
- 合法复杂 SQL 被误拒：记录拒绝类别，先用显式支持列表控制范围，后续按真实案例扩展 inspector 或评估 SQL parser。
- 元组语义被破坏：模型、provider、inspector 和测试全部使用元组或 map，不使用独立集合授权。
- provider 不稳定导致误放：provider 失败一律 fail-closed，并增加超时和异常测试。
- 通用 MCP 被 Apboa 逻辑污染：Apboa 逻辑只存在于 provider 和默认配置中，核心层只依赖通用接口。
- 非指标查询被影响：受保护资源检测未命中时保持原流程，并加入非指标回归测试。
- `user_id` 可伪造：明确列为非目标，由外部身份和网关体系解决。
- 预检工具被误当成边界：工具描述、ask-data 文案和服务端执行链路都强调实际 SQL 工具重复校验。

## 推荐落地策略

推荐采用“插件化 provider + 保守 SQL inspector”的第一版方案：

- 不从第一版开始引入大型 SQL parser 依赖，降低依赖、安全、许可证和多方言风险。
- 不做 SQL rewrite 或行级过滤自动注入，避免改变 SQL 语义。
- 用小而明确的 SQL 子集先满足安全要求，对不确定 SQL 直接拒绝。
- 通过配置和 provider 隔离 Apboa 默认实现，保持 MCP 通用性。
- 先打通 `execute_sql`、`explain_query`、`analyze_query_indexes` 的统一 enforcement，再扩展调试工具。

如果上线后发现大量合法 SQL 因 inspector 能力不足被拒绝，再基于真实样本评估引入 SQL parser 或扩展 AST 级检查。

## 验收标准

- MCP 服务端成为指标权限强制边界，不依赖 Agent 自觉。
- `execute_sql`、`explain_query`、`analyze_query_indexes` 均接入同一套指标权限校验。
- 指标 SQL 缺少 `permission_domain=metric` 时被拒绝。
- 指标 SQL 缺少 `user_id` 时被拒绝。
- 指标 SQL 缺少 `metric_scopes` 或等价元组声明时被拒绝。
- 声明元组不在 provider 授权范围内时被拒绝。
- SQL 证据与结构化声明不一致时被拒绝。
- 无法可靠判断的指标 SQL fail-closed 拒绝。
- provider 异常、超时、缺失或返回异常数据时 fail-closed 拒绝。
- 授权范围使用 `(quota_id, quota_scene)` 元组或等价 map，不存在独立集合交叉误放。
